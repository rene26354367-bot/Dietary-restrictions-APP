# 飲食營養 APP 專案 - 變更日誌 (CHANGELOG)

## [v5.3] - 2026-05-02 (本次更新)
### Added (新增)
- **High-Signal Image Pipeline**：實作前端影像前處理管道。
    - **動態解析度**：依據裝置硬體 (hardwareConcurrency) 自動切換 1800px 或 2400px 上限，防止低階機閃退。
    - **背景解碼**：使用 `createImageBitmap` 替代 `FileReader`，確保大圖處理時 UI 不卡頓。
    - **高品質重採樣**：強制開啟 `imageSmoothingQuality: 'high'` 並套用 `contrast(1.15)` 強化邊緣訊號。
- **Sanity Check UI**：在 `AddFood.tsx` 加入校驗結果顯示區。
    - **警告系統**：當偵測到熱量不一致或數據異常時，顯示顯眼的黃色警告圖示與診斷訊息。
    - **信任標籤**：通過邏輯驗證的數據會標註「數據已通過營養學邏輯驗證」綠色標籤。

### Changed (變更)
- **Parser 邏輯升級 (v5.3)**：
    - **三層關鍵字匹配**：建立「精確、OCR 容錯、特徵錨定」三級 ALIASES，解決「熟量」、「kca1」等掃描雜訊。
    - **條件式比例校驗**：僅在雙欄標籤下啟用，自動偵測並交換錯位的「每份/每100g」數據。
- **Sanity Check 核心**：引入營養學硬性約束。
    - 熱量守恆校驗：`P*4 + F*9 + C*4` 落在標示熱量的 ±20% 內。
    - 總重量約束：單純營養素加總 `P+F+C` (每 1g 基準) 不可超過 1.1g。
    - 飽和脂肪誤抓偵測：若脂肪占比極端且包含「飽和」關鍵字，自動標註警告。

### Fixed (修正)
- **[P0] 100g 換算殘留錯誤**：完全重新設計了 Parser 的數據交換與縮放邏輯，確保不論 OCR 讀到哪一欄，UI 都能正確顯示 100g 基準。
- **低階手機相容性**：透過 Pipeline 降低了記憶體峰值佔用，解決了部分 Android 手機拍照上傳失敗的問題。

## [v4.2] - 2026-05-02
### Added (新增)
- **俗名別名系統**：新建 `aliases.json`（~50 筆俗名 → 官方名映射），支援「小黃瓜→胡瓜」、「牛奶→鮮乳」等別名查詢。
- **別名提示顯示**：搜尋結果新增 `matchedAlias` 欄位，前端顯示灰字提示使用者「官方名（俗名）」對應關係，減少混淆。
- **OCR 雙欄位識別增強**：支援 stacked header 排版（例如「營養標示」下方換行再列「每一份量」），使用 `_detectColumnLayout()` 動態檢測排版。
- **OCR 多行數值抽取**：實裝 `consumed` 集合追蹤，防止糖分標籤 (sub-label) 誤用碳水化合物的克數。

### Fixed (修正)
- **[P0] OCR 份量換算殘留錯誤**：根本原因是雙欄位標籤（每份/每100g）識別失敗，現已完全修正。
  - 修正 `_detectColumnLayout()`：從僅支援 inline 格式，擴展至 stacked（換行型）與 single 格式。
  - 修正 `_detectBaseWeight()`：改從完整 normalized 文本偵測（之前在 anchor 後的片段文本），避免「每一份量」被截斷。
  - 修正條碼號污染：加入 `MAX_PER_SERVING = 9999` 濾鏡，排除如「199 013304」的條碼數字誤認為鈉含量。
  - 修正糖分干擾：實裝 `SKIP_OVER` 機制 + `consumed` 集合，允許碳水化合物跳過相鄰的糖分標籤行。
  - 測試驗證：新增 3 個 v5.1 單元測試（3/3 ✅）+ 14 張測試圖片批量驗證，顯著改善識別精度。

- **搜尋結果排序最佳化**：別名展開的官方食材 (alias-matched) 優先於直接名稱匹配。
  - 改動 `AppEngine.js` 的 `searchFood()`：分離 `directResults` 與 `aliasResults`，排序為 `[customFoods, aliasResults, directResults]`。
  - 效果：搜「牛奶」現在第 1-2 筆直接顯示鮮乳類（帶「牛奶」灰字），而非 22 個「義美牛奶小泡芙」等加工品。

- **別名對應錯誤**：
  - `"牛奶": "牛乳"` → 改為 `"鮮乳"`（DB 無「牛乳」項目，有「全脂鮮乳」「低脂鮮乳」等）。
  - `"白米": "白米飯"` → 改為 `"白飯"`；新增 `"白米飯": "白飯"`（DB 項目名稱是「白飯」不是「白米飯」）。

### Challenges (遇到的問題 & 解決過程)
- **fullName 搜尋誤判**：初期嘗試在 `db_manager.searchFood()` 中搜 `fullName` 欄位，導致「千島沙拉醬」(fullName 含「小黃瓜」作為食材成分) 被意外返回。決議：放棄 fullName 搜尋，改由 aliases.json + 二層搜尋架構解決。
  
- **別名資料庫對應理論**：「牛奶」的官方名是什麼？使用者期望搜「牛奶」找到"鮮乳"，但資料庫實際項目名稱隨廠商而異（全脂鮮乳、高脂鮮乳、低脂鮮乳等）。初期錯誤對應到不存在的「牛乳」，經診斷 DB 內容後改正。

- **搜尋排序決策**：直接名稱匹配 (「牛奶」在食物名中) vs 別名展開的官方食材，哪個應優先？確定為別名優先，因使用者輸入別名時的意圖是"找官方食材"，不是"找巧合包含這個詞的加工食品"。實裝時從簡單的 `allResults.push()` 改為區分 `directResults` 與 `aliasResults` 並排序。

### 驗證狀況
- **API 端點測試**：隨機抽樣 10 個俗名搜尋，10/10 成功通過（涵蓋別名單/雙層展開、直搜無別名等場景）。
- **OCR 修正驗證**：示範圖片 `2016-06-08 at 10-09-46.jpg` 從誤認的 220.1 kcal (每份) 改正為 244.6 kcal (每100g)。批量 14 張圖片掃描，顯著減少誤差。

## [v4.1] - 2026-05-01
### Added (新增)
- **實體相機掃描**：`AddFood.tsx` 整合 `capture="environment"`，支援直接拍照辨識營養標示。
- **Google Cloud Vision 串接**：後端 `server.cjs` 完成圖片辨識 API 對接，具備 Base64 圖片處理能力。
- **100g 強制標準化**：全系統 OCR 輸出統一以 100g 為基準，自動處理份量換算。

### Fixed (修正)
- **脂肪解析誤差**：修正總脂肪會誤抓飽和脂肪的 Bug（引入負向先行斷言）。
- **跨行數字抓取**：`NutritionParser` 現在能辨識關鍵字與數值不在同一行的排版。
- **數字斷裂修正**：自動接合 OCR 產生的空格數字（例如 `1 5 0 . 5` -> `150.5`）。
- **雙欄位防呆**：當標籤有雙欄（每份/每100g）但 OCR 漏看一欄時，系統能智能判定基準並正確換算。
- **UI 邏輯衝突**：將 OCR 原始文字與手動輸入框分離，避免干擾使用者操作。
- **靜默歸零 Bug**：修復解析失敗時會顯示 `0.0` 的問題，改為回傳空值以誠實反映辨識現況。

### Challenges (遇到的問題)
- **GCP 結算限制**：Cloud Vision API 必須啟用 Billing 帳戶才能運作（即使在免費額度內）。
- **環境變數干擾**：`.env` 中的引號會導致 API Key 辨識失敗。
- **[待處理 P0] 100g 換算殘留錯誤**：雖然 UI 顯示 100g，但數值仍維持原始「每份」數據（例如 90g 的熱量），換算係數未正確套用。需檢查 `NutritionParser` 回傳的 `isPer100` 標記与 `divisor` 邏輯。

---
**決策紀錄 (Key Decisions):**
1. **1g 基準原則**: 所有資料統一以 1g 為儲存基準，確保換算精度。
2. **人性化份量**: 根據性別預設「一碗/一份」的克數係數，提升用戶體驗。
3. **體驗優先 OCR**: 初期策略採用 Cloud Vision API 以確保精準度，同時預留 On-device OCR 擴充空間。
4. **零成本營運**: 優先選擇開源 API (Open Food Facts) 與本地運行邏輯，避免失控的雲端費用。
